# Add Hosted Inbox Upload And Ingest API

## Parent

Clerk Practice Intelligence Integration PRD.

## What to Build

Add the backend path for hosted project evidence intake. An authenticated user uploads one or more files through the Clerk API; the backend stores the originals in Supabase Storage under the active project's SiteWise `_inbox/` path, records workspace file metadata in Postgres, and runs the existing ingest pipeline so extracted text, chunks, and embeddings become searchable project evidence.

Uploaded files must land in `_inbox/` by default. Ingest happens automatically after upload — the user does not run a separate ingest command. Sorting files into lifecycle folders is a later workflow (`CPI-015`).

## Acceptance Criteria

- [x] Clerk can store uploaded project file bytes in Supabase Storage scoped to the authenticated user's project.
- [x] Each upload is stored at a logical SiteWise path under `{project.workspace_path}/_inbox/` (nested relative paths preserved when supplied).
- [x] Postgres records exist for uploaded inbox files with content hash, size, storage key, ingest status, and workspace path.
- [x] `POST /projects/{project_id}/inbox/upload` accepts multipart file upload(s) and enforces project ownership at the API boundary.
- [x] After a successful upload, the backend triggers ingest from stored bytes (not from server `DATA_DIR`) using the existing extract/classify/chunk/embed pipeline.
- [x] Ingest calls `parse_document_metadata` (`CPI-017`) with filename, workspace path, and bounded first-page preview so `document_metadata` includes register fields: document number, title, revision, discipline/category, canonical filename, and confidence.
- [x] Ingest creates or updates `source_documents` and `document_chunks` rows tied to the active project slug.
- [x] Re-uploading identical content is idempotent or safely rejected according to content hash behaviour.
- [x] Supported extensions match current ingest settings (PDF, DOCX, Markdown at minimum).
- [x] Unsupported or empty files return a clear API error without partial storage corruption.
- [x] Backend tests cover auth, project scoping, inbox path assignment, and ingest persistence without live OpenAI or Supabase calls in the fast suite.
- [x] Deployment docs or env examples note required Supabase Storage bucket configuration and backend-only credentials.

## Agent Brief

**Category:** enhancement
**Summary:** Store uploaded project files in Supabase Storage under `_inbox/` and ingest them into retrieval tables.

**Current behavior:** Project evidence enters Clerk only through offline CLI ingest of the local `data/` corpus. There is no hosted upload path.

**Desired behavior:** The backend accepts authenticated inbox uploads, persists binaries to object storage, and auto-ingests into `source_documents` / `document_chunks` using the existing pipeline.

**Key interfaces:** Supabase Storage (service role from backend), workspace file persistence model, `POST /projects/{id}/inbox/upload`, ingest pipeline entry point for storage-backed bytes, project slug scoping.

**Acceptance criteria:** Upload defaults to `_inbox/`, ingest runs automatically, hashes and ingest status are recorded, and tests cover scoping and idempotency.

**Out of scope:** Browser drag-and-drop UI (see `CPI-014`), Sort Files classification and moves (see `CPI-015`), register table UI (see `CPI-016`), email intake, zip bulk import.

## Implementation Notes

- `POST /projects/{project_id}/inbox/upload` — multipart `files` + optional `relative_path` form field.
- `workspace_files` table and Alembic migration `004_workspace_files.py`.
- Storage via `app/storage/project_files.py` (service-role Supabase client); bucket config `SUPABASE_STORAGE_BUCKET` in `.env.example`.
- Hosted ingest entry point: `ingest/hosted.py` (temp file → existing pipeline).
- Upload orchestration: `app/inbox/service.py`, path helpers: `app/inbox/paths.py`.
- Tests: `tests/inbox/test_paths.py`, `tests/inbox/test_upload.py`.

## Design Notes

- Preserve the SiteWise authority stack: uploaded inbox files are project evidence, not platform knowledge.
- Keep service-role Storage access backend-only; the browser uploads through FastAPI, not directly to Supabase Storage with privileged keys.
- Logical `workspace_path` remains the user-facing filing identifier even when bytes live in object storage.
- Ingest failures should leave the stored binary intact and mark ingest status as failed so the UI can retry.
