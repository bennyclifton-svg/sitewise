# Use Repository Panel As Inbox Upload Drop Zone

## Parent

Clerk Practice Intelligence Integration PRD.

## What to Build

Make the far-right document repository panel the project cockpit's upload surface. When a user drags one or more files onto `DocumentRepositoryPanel`, Clerk uploads them through the hosted inbox API, shows upload and ingest progress, and lists the resulting inbox evidence in the repository once indexing completes.

This is step one of hosted intake: **upload + auto-ingest into `_inbox/`**. It does not run Sort Files. The user classifies and files documents later from the Document Intake workflow (`CPI-015`).

## Acceptance Criteria

- [x] The far-right panel (`DocumentRepositoryPanel` in `ProjectShell`) is the designated upload drop zone for the project cockpit.
- [x] Dragging files over the repository panel shows a clear drop affordance (highlight/border/copy) indicating files will upload to `_inbox/`.
- [x] Dropping supported files uploads them via `POST /projects/{project_id}/inbox/upload` with the user's Supabase bearer token.
- [x] Multi-file drop uploads all files in one interaction where the API supports batch upload.
- [x] While uploads or ingest are in progress, the panel shows per-file or aggregate progress and disables accidental duplicate submission.
- [x] After ingest completes, uploaded files appear in the repository list once evidence refresh runs (full register table columns ship with `CPI-016`).
- [x] Failed uploads or ingest failures show an inline error with retry guidance; successful files are not lost when another file in the batch fails.
- [x] The empty repository state instructs users to drag files into the panel to add evidence to `_inbox/`.
- [x] Upload works from the project cockpit route without navigating away or opening a separate modal-first flow.
- [x] Typecheck and lint pass; manual verification covers drag-drop upload on a hosted or local backend with auth.

## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement  
**Summary:** Turn the far-right repository panel into a drag-and-drop inbox upload zone wired to the hosted upload API.

**Current behavior:** The repository panel lists indexed evidence snippets only. There is no upload affordance and the empty state says evidence appears after ingestion with no user action.

**Desired behavior:** Users drag files onto the right panel; Clerk uploads to `_inbox/`, ingests automatically, shows per-file status, and refreshes the evidence list when ingest completes.

**Key interfaces:**

- `DocumentRepositoryPanel` — accept `projectId`, drag/drop handlers, upload progress UI, disabled state while busy
- `api.uploadInboxFiles(projectId, files)` — `FormData` multipart to `POST /projects/{id}/inbox/upload`; bearer token via existing auth helper
- `ProjectCockpitPage` — pass `projectId`, `onEvidenceRefresh` callback after successful uploads
- Supported extensions: `.pdf`, `.docx`, `.md` (match backend `INGEST_SUPPORTED_EXTENSIONS`)

**Acceptance criteria:**

- [x] Panel is sole cockpit upload surface (no centre/workbench upload)
- [x] Drag-over affordance + empty-state copy mentions `_inbox/`
- [x] Multi-file batch upload in one request
- [x] Progress/disabled state during upload+ingest
- [x] Inline errors for failed files; refresh evidence after batch
- [x] `pnpm tsc --noEmit` and `pnpm lint` pass

**Out of scope:** Sort Files workflow (`CPI-015`), register table columns (`CPI-016`), nested inbox folder picker UI, direct Supabase Storage access from browser.

## Implementation Notes

- `api.uploadInboxFiles` in `src/lib/api.ts` — multipart `FormData`, extended timeout for ingest.
- `DocumentRepositoryPanel` — drag/drop, click-to-browse on empty dashed area, upload status list, `SUPPORTED_INBOX_EXTENSIONS` client validation.
- `ProjectCockpitPage` — `refreshEvidence()` wired as `onUploadComplete`.

## Design Notes

- Do not add a competing upload surface in the centre workbench or chat bar for this slice; the repository panel owns upload.
- Match existing cockpit styling (Tailwind/shadcn) rather than copying Practice's local filesystem assumptions.
- Prefer refreshing project evidence/document list after upload rather than optimistic-only UI that hides ingest failures.
