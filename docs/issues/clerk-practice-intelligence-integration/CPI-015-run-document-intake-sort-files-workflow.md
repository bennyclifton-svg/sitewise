# Run Document Intake Sort Files Workflow

## Parent

Clerk Practice Intelligence Integration PRD.

## What to Build

Implement the hosted **Document Intake** workflow (Practice Intelligence "Sort Files") as the second step after inbox upload. After files sit in `_inbox/` (`CPI-013`, `CPI-014`), the user opens the Document Intake tile on the control board and clicks **Run Sort Files** (or equivalent label). Clerk inspects inbox entries, classifies only confident matches into known SiteWise lifecycle folders, leaves unclear files in `_inbox/`, writes a versioned intake manifest draft, and surfaces a result table in the cockpit.

This workflow performs **logical filing** in hosted mode: update workspace paths and move objects in Supabase Storage — not local disk moves.

## Acceptance Criteria

- [x] `POST /projects/{project_id}/workflows/sort-files` (or equivalent) runs Sort Files for the active project only.
- [x] The SiteWise three-overlay gate blocks the workflow when archetype, user role, or state is missing, consistent with other phase-gate workflows.
- [x] The workflow inspects files under `{project.workspace_path}/_inbox/` including nested inbox paths.
- [x] Confident classifications move files to known SiteWise destination folders (for example `03-design/architect/`, `04-planning-and-authorities/`, `05-procurement/`) by updating storage location and workspace metadata.
- [x] Confident moves rename files to Practice canonical format `{documentNumber} - {title} Rev {revision}.{ext}` when `parse_document_metadata` confidence is not `low` (`CPI-017`).
- [x] Sort Files upserts register rows (document number, title, revision, category) for moved and already-filed files, equivalent to Practice `buildDocumentRegisterRowsFromSortResult`.
- [x] Unresolved, skipped, refused, and already-filed outcomes follow `data/skills/systems/document-intake-register-system.md` guardrails (no delete, no overwrite, no moves outside the active project).
- [x] The workflow writes an intake manifest as a versioned draft artefact (for example `_inbox/intake_manifest_vNN.md`) with outcome counts and per-file detail.
- [x] Re-ingest or path-update runs after successful moves so retrieval reflects filed locations.
- [x] Workflow trace events stream to the project chat thread like Create PMP.
- [x] The Document Intake tile on `ProjectControlBoard` exposes a working **Run Sort Files** action when inbox files exist and the gate passes.
- [x] The centre workbench shows a Sort Files result summary (inspected, moved, unresolved, refused counts and per-file rows), ported from Practice's result table pattern.
- [x] Backend tests cover classification outcomes, guardrails, gate blocking, and manifest persistence without live model calls where possible.

> *This was generated by AI during triage.*

**Category:** enhancement  
**Summary:** Add hosted Sort Files workflow after inbox upload, with manifest draft and cockpit result UI.

**Current behavior:** Document Intake appears as a control-board tile but is not implemented. Practice Intelligence sorted files on the local filesystem via the Node storage adapter.

**Desired behavior:** User uploads to `_inbox/` via the repository panel, then explicitly runs Sort Files from Document Intake to classify and file documents, with auditable manifest output.

**Implementation plan:**

1. **`backend/app/intake/classifier.py`** — Port v1 inbox package → lifecycle folder map from `document-intake-register-system.md` + `INBOX_FOLDER_DISCIPLINE_LABELS`; filename fallbacks (CC-A-*, E##, H-*, etc.); return `None` when not confident.
2. **`backend/app/intake/sort_service.py`** — List `workspace_files` under `{workspace}/_inbox/`; skip `intake_manifest_v*.md`; classify; guardrails (no delete, no overwrite, project boundary); move via storage copy+delete; canonical rename when metadata confidence ≠ `low`; purge old `source_documents` row; re-ingest at new path; build register metadata.
3. **`backend/app/workflows/sort_files.py`** — Gate → sort → manifest markdown → `create_draft_artifact(workflow_type="sort_files")` → trace to chat thread (mirror Create PMP).
4. **`POST /projects/{project_id}/workflows/sort-files`** — Return counts + per-file rows + draft + trace.
5. **Frontend** — Wire Document Intake tile: Run Sort Files button when inbox files exist + gate passes; `SortFilesResultPanel` with inspected/moved/unresolved/refused counts and per-file table; add `sort_files` to `IMPLEMENTED_TILES`.

**Key interfaces:** `classify_inbox_file`, `run_sort_files_workflow`, `move_project_file`, `list_inbox_workspace_files`, `SortFilesResponse` schema, `ProjectControlBoard` + `SortFilesResultPanel`.

**Reference material:** `data/skills/systems/document-intake-register-system.md`, `backend/ingest/document_metadata.py`, `app/workflows/create_pmp.py`, Practice `sortInboxFiles` / `SortFilesResultPanel` (behavioural contract only).

**Test plan:** Fast pytest — classifier destinations from inbox packages and drawing prefixes; refused when destination hash differs; skipped manifests; gate blocks; manifest version increments; no live model calls.

**Out of scope:** Document-register row drafting mode (`sort-and-draft-register-rows`), invoice/variation cost processing, email auto-file, OCR/drawing deep parse, CPI-016 register table UI.

## Design Notes

- Upload and Sort Files must remain separate user actions: dropping files does not auto-file into lifecycle folders.
- Port classifier behaviour from Practice; implement storage-backed moves in Python rather than calling the Node adapter.
- Re-run metadata extraction after move/rename so register table columns stay accurate (`CPI-016`).
