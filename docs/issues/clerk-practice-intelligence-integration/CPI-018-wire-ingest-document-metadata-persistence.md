# Wire Ingest Document Metadata Persistence

## Parent

Clerk Practice Intelligence Integration PRD — extracted from `CPI-013` acceptance criterion on register metadata at ingest time.

## What to Build

Every document ingested through the existing pipeline (CLI corpus ingest today; hosted inbox upload in `CPI-013`) must call `parse_document_metadata` from `CPI-017` and persist register-oriented fields on `source_documents.document_metadata`.

This is the **integration slice** between the metadata extraction engine and ingest persistence. It does not implement upload/storage (`CPI-013`) or the register table UI (`CPI-016`).

## Acceptance Criteria

- [x] Ingest persistence calls `parse_document_metadata` with `file_name`, `filed_path` (workspace-relative path), optional bounded `preview_snippet`, and optional `source_path` (same as filed path when no separate source).
- [x] `document_metadata` JSONB includes register fields: `document_number`, `title`, `revision`, `discipline`, `canonical_file_name`, `metadata_confidence`.
- [x] Legacy drawing fields (`drawing_number`) remain populated when they carry the same value as `document_number` for backward compatibility with existing register retrieval.
- [x] Preview snippet is derived from extracted content (title-block region for drawings, bounded first-page text for PDF/DOCX/Markdown) without re-reading the file at persist time.
- [x] Classification-specific metadata (procurement stage, platform scope, etc.) is preserved and merged with register fields; register fields do not overwrite unrelated keys.
- [x] Pytest covers metadata merge in persist layer with mocked extraction output (no live DB/OpenAI in fast suite).

## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement  
**Summary:** Call `parse_document_metadata` during ingest persist and write register fields to `source_documents.document_metadata`.

**Current behavior:** Ingest stores filename, classification keys, and coarse drawing identity (`drawing_number`, `revision`, `title` from filename-only parsing). Register-grade document number, discipline/category, canonical filename, and confidence are not populated.

**Desired behavior:** After extraction, ingest builds a bounded preview snippet from the extracted document, calls `parse_document_metadata`, and merges the result into `document_metadata` alongside existing classification and platform keys. Re-ingest updates metadata when content changes.

**Key interfaces:**

- `parse_document_metadata(file_name, filed_path, preview_snippet?, source_path?)` → `DocumentMetadata` (`CPI-017`)
- `_merged_metadata(plan, extracted)` in ingest persist — extend to invoke parser and map output keys
- `ExtractedDocument` — preview snippet derived from normalized content / title-block section
- `SourceDocument.document_metadata` JSONB — authoritative store for register API (`CPI-016`)

**Acceptance criteria:**

- [x] Register fields persisted on every successful ingest upsert
- [x] Preview snippet passed when available from extraction
- [x] `drawing_number` kept in sync with `document_number` when present
- [x] Persist-layer pytest verifies field mapping without network

**Out of scope:** Supabase Storage upload path, frontend register table, Sort Files workflow, re-parsing metadata in the API layer.

## Implementation Notes

- `ingest/persist.py` — `_preview_snippet`, `_register_metadata`, `_merged_metadata`.
- Parse failures are logged and do not block ingest (best-effort empty register fields).
- Tests: `backend/tests/ingest/test_persist_metadata.py`.

## Design Notes

- `filed_path` should use the ingest `relative_path` (SiteWise workspace path).
- When confidence is `low`, canonical filename in metadata should remain the original upload filename per Practice rules.
- Do not block ingest on metadata parse failure — log and persist best-effort empty register fields.
